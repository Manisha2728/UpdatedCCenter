Defines various environment variables.
If the properties based on the executable application are not enough to differentiate two different EHR Agent scenarios, the environment variables are used. In this case, a different environment variable will be set on the client computers working in different scenarios.

  • IIS(CVA)
Context properties define the way that User and Patient data are retrieved from an EHR

Full namespace and assembly name are required in Non-CCOW scenario.

Enter the UI Automation configuration files full path including the file extension.

User mapping is needed when the user context must be obtained from the EHR display, and the user`s display name is not identical to the username in the Active Directory.

User Mapping Properties Services to restart

  • IIS(CVA)
Configuration for user mapping

Enter the mapping file's full path including the file extension, such as \MachineName\FolderName\UserMapping.xml.
The file path must be accessible from the application servers.

If selected, the user context will not be resolved by name. The role of the user will be used by the Agent Hub to determine the correct permissions for hosted applications.

If the displayed user is listed multiple times in the mapping file, use the logged-in user by default.

User Facility Identification Services to restart

  • IIS(CVA)
User Facility Identification detects the facility mapped to the EHR instance intercepted by the EHR Agent.
For more information about the functionality associated with this capability, see the EHR Agent Functional Specification document.

Defines the root Identifier of the facility, according to [dbmVCDRData].[Common].[Organization].[Id_Root] .

Defines the extension Identifier of the facility, according to [dbmVCDRData].[Common].[Organization].[Id_Extension].

The active EHR instance facility is identified by the unique features of the EHR instance.
To distinguish between EHR instances associated with different facilities, configure one of the following:
- If the EHR instance has a unique EXE file name , Window title, or URL per facility, configure only the Facility ID Root and Extension fields.
- If the EHR instance EXE file name and Window title is not unique per facility, configure each EHR instance with the relevant environment variable name and value, as well as the Facility ID Root and Extension fields.

More options Services to restart

  • IIS(CVA)

Defines whether EHR Agent will exclude dash character from the MRN, which recieved in the CCOW context.
NOTE: This funcionality is availible only when Agent Hub is CCOW Participant or CCOW Context Manager.
By default, this functionality is enabled.
Default: True

Defines whether EHR Agent will display the Badging only during the patient's first day of admission.
By default, this functionality is disabled. This means that by default the EHR Agent will display the Badging in all cases where there is a delta in the configured attention time frame.
Default: False

Defines a comma-separated list of Profiles that can use this EHR instance. If empty, all profiles can use it.
The Profile defines various installation attributes that determine how Agent Hub will be installed on the client machine.
Each Profile has a unique sub-folder under: \\Web App Server\EHRAgentSetup\.
For example: Default Profile, CCOW, CCOW Shared Desktop.
Default: Empty

Defines Direct Address associated to the EHR instance.
When users send Direct messages from ClinicalView Agent, this address will be the user's sending address.
Example: Facility.name@location.direct.com
Default: Empty

Define suffix for a group of direct addresses.
Should only use in case the EHR using privates direct addresses for his providers and routed the direct messages also to dbMotion
Example: @location.direct.com
Default: Empty

Defines the endpoint URL used to report Sending/Receiving of CCDA documents over Direct.
Default: Empty

Define the source system name as exist in the CDR common.device.devicename.
Note: This field should be configured only if two or more EHR instances using the same OID in "Assigning Authority for display" and the reporting process should find only the relevant patient for the EHR instance.
Default: Empty

Defines the interface used to report Sending/Receiving of CCDA documents over Direct.
Note: This field is mandatory if an endpoint is configured.
SCM: SCM interface (reporting to CPM).
TW: TW interface (reporting to AAP).
Default: Empty

Defines thumbprint of the certificate for the configured endpoint in the field: Meaningful Use Reporting Endpoint.
Note: This field is mandatory if an endpoint is configured and SCM is the reporting interface.
Default: Empty

Define the Unity service user that will be used when reporting to TW.
Unity service user should be configured for the relevant TW environment: TEST / PROD.
Note: This field is mandatory when TW is the reporting interface.
Default: Empty

Define the Unity service Password that will be used when reporting to TW.
Unity service Password should be configured for the relevant TW environment: TEST / PROD.
Note: This field is mandatory when TW is the reporting interface.
Default: Empty

Define dbMotion application name as configured in Unity. Application name should be configured for the relevant TW environment: TEST / PROD. Note: This field is mandatory when TW is the reporting interface.
Default: Empty

Define TW Community OID as configured in TW, for the dbMotion - TW integration.
The value should be provided from TW resource.
Note: This field is mandatory when TW is the reporting interface.
Default: Empty

Defines a CCOW parameter with an added suffix (for example, the syntax will be Patient.Id.Mrn.Suffix) that is used to identify an EHR instance.

Configure this parameter when there is a need to differentiate between EHR instances, when all instances have the same values for the following parameters (meaning that the system cannot differentiate between them):
- Same EHR (Application)
- URL (for Web application)
- Executable file name (for Desktop application)
- Application Title
- Application Version
- Environment Variable (or cannot create it)
- Context Type
- Participant Name
- Passcode
- Same Supported profiles

This configuration provides an additional method for organizations to differentiate between the instances of the same EHR. When the instance is identified, the relevant product functionality will apply per instance, as usual.

- Delta View can be configured per instance
- Unique Patient Assigning Authority can be configured per instance
- Send to My EHR endpoint can be configured per instance

For the implementation of this functionality, the following conditions apply:
- It applies only with context interception using CCOW.
- Each EHR instance is configured with a separate Properties Package.

Limitation:
This configuration is ignored for an Instance with a Property Package that has more than one configured Patient Assigning Authority.
